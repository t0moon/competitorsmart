# 设计评审

## 何时使用此技能

当你需要：
- 组织和主持设计评审会议
- 提供结构化的设计反馈
- 建立高质量的设计评审流程
- 在发布前确保设计质量

## 核心原则

### 1. 价值优先的反馈层次
使用层次结构综合设计反馈：首先是价值，然后是易用性，最后是愉悦感。

### 2. 需要项目赞助人
有效的设计评审需要项目"赞助人"在实时演示中提供关于"为什么"和核心概念的高层反馈。

### 3. 大局优先于细节
有效的设计反馈必须从高层目标和"大局"感受开始，然后再深入具体技术细节。

## 专家洞察

### Julie Zhuo
> "首先最重要的是解决：这个东西实际上有价值吗？它在解决问题吗？它在正确地完成工作吗……然后一旦我们做到了，让我们专注于下一层，我认为是易用性……最后，如果它有价值，易于使用，然后是……"

**核心洞察**：使用层次结构综合设计反馈：首先是价值，然后是易用性，最后是愉悦感。

**如何应用**：
- 在核心价值主张得到验证之前，忽略关于美学或小细节的反馈
- 将反馈分类到"价值"、"易用性"和"愉悦感"桶中以确定优先级

### Karri Saarinen
> "我仍然管理设计团队，所以我确实每周看到一些设计，然后，或者其他联合创始人之一或产品负责人，我们基本上是项目的赞助人。所以我们负责审查工作。所以我们可能只是开个会，过一遍，好的……"

**核心洞察**：有效的设计评审需要项目"赞助人"在实时演示中提供关于"为什么"和核心概念的高层反馈。

**如何应用**：
- 为每个重大项目指定一位创始人或高级领导者作为"赞助人"来监督质量
- 通过实时演示而非静态幻灯片进行评审，以看到真实交互

### Dmitry Zlokazov
> "基本上公司的创始人仍然审查100%正在发布的屏幕，你在应用中看到的一切都通过了这个审查。有很多眼睛在审视这个并思考我们需要考虑的所有可能的边缘情况和细微差别……"

**核心洞察**：通过让创始人或高级领导者在发布前审查每一个面向用户的屏幕来保持高质量标准。

**如何应用**：
- 实施一个100%发布屏幕都由领导层审查的评审流程
- 将评审重点放在边缘情况和细微差别上以确保客户满意度

### Geoff Charles
> "我认为我们现在关联的是，路线图上的任何大石头都需要被带到产品评审流程中，我和设计负责人在场并给出反馈，但它需要以这样的方式构建：你明确要求你寻求什么类型的反馈……"

**核心洞察**：围绕特定的反馈请求和明确强调的风险和权衡来构建设计评审。

**如何应用**：
- 只将"大石头"或高风险决策带到正式评审
- 要求演示者明确说明他们寻求什么类型的反馈
- 使用Figma原型的Loom演练来激励团队并锚定愿景

### Jessica Hische
> "最好的事情是看看那里有什么，真正能够理解什么不起作用以及你的目标是什么……在考虑细枝末节之前总是先想大局，因为有时候人们认为……他们会向我扔一堆细枝末节的东西，但那是因为他们……"

**核心洞察**：有效的设计反馈必须从高层目标和"大局"感受开始，然后再深入具体技术细节。

**如何应用**：
- 在评论具体元素如字母或颜色之前，先问"整体上困扰我的是什么？"
- 使用"模糊眼睛"来看品牌的整体连贯性，而不是关注单个像素

## 常见错误

- 一上来就关注细节而不是价值
- 没有明确的评审层次结构
- 使用静态幻灯片而非实时演示
- 不明确说明需要什么类型的反馈
- 每个屏幕都评审导致瓶颈

## 关键战术

| 战术 | 说明 |
|------|------|
| **价值-易用性-愉悦层次** | 按此顺序优先考虑反馈 |
| **项目赞助人制** | 为每个项目指定高级领导者负责质量 |
| **实时演示评审** | 通过实时交互而非静态图片评审 |
| **模糊眼睛法** | 先看整体连贯性再看细节 |

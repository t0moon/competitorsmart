# 可用性测试

## 何时使用此技能

当你需要：
- 验证产品设计的可用性
- 测试原型的用户体验
- 在写生产代码前验证核心价值
- 观察用户真实行为

## 核心原则

### 1. 先假后真
使用"假"版本的产品（绿野仙踪或烟雾测试）在写生产代码前验证核心价值主张。

### 2. 微观测试价值巨大
微观层面的评估性研究和可用性测试可以通过小的战术优化产生巨大的业务价值。

### 3. 野外观察
培养产品直觉需要在野外观察人类与技术的交互，而非只在受控的实验室环境中。

## 专家洞察

### Itamar Gilad
> "一开始你假装它，你做一个假门测试，你做一个烟雾测试，绿野仙踪测试。我们在标签收件箱中用了很多这些，顺便说一下，最早的版本之一实际上是我们向人们展示标签收件箱在工作。但它并不是真正的 Gmail，只是一个 HTML 的门面和..."

**核心洞察**：使用产品的"假"版本（绿野仙踪或烟雾测试）在写生产代码前验证核心价值主张。

**如何应用**：
- 运行"绿野仙踪"测试，人工在门面后执行自动化任务
- 使用"假门"测试来测量用户对不存在功能的意图和点击率

### Judd Antin
> "在微观层面，那里有太多的业务价值可以挖掘...我们改变了按钮上的文字，在我们出色的内容设计的帮助下...我们基本上改变了七个字符，为 Airbnb 赚了数百万美元，因为我们发现的真的很简单。就像'嘿，这个..."

**核心洞察**：微观层面的评估性研究和可用性测试可以通过小的战术优化产生巨大的业务价值。

**如何应用**：
- 不要把可用性测试当作"初级"工作；它对业务指标有高杠杆作用
- 寻找可能阻塞转化漏斗的"吓人"或令人困惑的 CTA

### Upasna Gautam
> "我计划了一个与用户的大型工作会议来做研究，或者做用户测试，突发新闻发生了，聚集全球编辑团队来做用户测试会议需要花很多时间和精力。当突发新闻发生时，他们必须把那个优先于..."

**核心洞察**：在像新闻这样的高压环境中进行用户测试需要极度灵活，因为用户总会优先处理突发事件而非研究会议。

**如何应用**：
- 为涉及繁忙利益相关者的研究会议建立缓冲和备份
- 准备好随时调整或重新安排

### Amjad Masad
> "我们看到产品经理构建，像我说的，一个应用的 v1 并实际出去和用户测试。我不能说出公司名字，但有一家上市公司用 Replit 测试了一个应用的 v1。显然在那种方式奏效后，他们把它带给工程师，他们..."

**核心洞察**：AI 工具让 PM 能够绕过工程瓶颈来创建功能原型进行真实世界的用户测试。

**如何应用**：
- 独立构建一个功能性 v1 来与用户验证想法，然后再放入官方工程路线图

### Bob Baxley
> "去观察人们在 Target 使用自助结账...然后去其他某个不那么好的杂货店观察，真正注意人们发生了什么...去看看 70 岁以上的人摸索芯片卡插入或看看某人试图弄清楚 Apple Pay。"

**核心洞察**：培养产品直觉需要在野外观察人类与技术的交互，而非只在受控的实验室环境中。

**如何应用**：
- 通过观察用户与竞品交互进行"现实检查"来看他们无偏见的行为和需求

## 常见错误

- 在验证价值前就写生产代码
- 把可用性测试当作初级工作忽视
- 只在受控实验室环境中观察用户
- 研究计划没有足够的灵活性

## 关键战术

| 战术 | 说明 |
|------|------|
| **绿野仙踪测试** | 人工在门面后执行"自动化"功能 |
| **假门测试** | 测量用户对不存在功能的点击意图 |
| **微观优化** | 关注按钮文字等小细节的高杠杆改进 |
| **野外观察** | 在真实环境中观察用户与技术的交互 |

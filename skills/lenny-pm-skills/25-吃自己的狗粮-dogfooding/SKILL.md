# 吃自己的狗粮

## 何时使用此技能

当你需要：
- 深入理解用户痛点
- 验证产品功能的实用性
- 培养团队的产品直觉
- 建立产品开发的真实感

## 核心原则

### 1. 成为自己的用户
团队必须成为产品的真实用户，而不仅仅是测试者。

### 2. 不发布对自己无用的功能
如果一个功能对团队自己没用，就不应该发布。

### 3. 日常深度使用
偶尔试用不够，需要每天深度使用产品。

## 专家洞察

### Maya Prohovnik
> "我一直在对那些没有自己做播客的产品团队成员大声疾呼，说'我真的不认为你能构建正确的东西'。如果他们一直和用户交流，看数据，但所有人一旦最终开始做自己的播客，他们就会说'我懂了'。"

**核心洞察**：Dogfooding是核心领导哲学，要求整个团队成为创作者，以深入理解用户痛点。

**如何应用**：
- 要求团队成员亲身体验用户的完整流程
- 不能只看数据和用户反馈，要有亲身体验
- 将dogfooding作为团队文化要求

### Michael Truell
> "从一开始，我们的产品开发流程就是关于dogfooding，每天深度使用这个工具。我们从不想发布任何对我们自己没用的东西。"

**核心洞察**：他们的主要产品开发引擎是每天"深度"使用自己的工具，这提供了构建有用AI功能所需的"真实感"。

**如何应用**：
- 将每天使用自己的产品作为开发流程的核心
- 用"对我们自己有用吗"作为发布标准
- 通过深度使用获得构建功能的真实洞察

## 常见错误

- 只在测试时用产品，不在真实场景使用
- 团队不是目标用户就放弃dogfooding
- 偶尔试用，没有养成日常使用习惯
- 只看表面功能，不深入体验完整流程

## 关键战术

| 战术 | 说明 |
|------|------|
| **强制体验** | 要求团队成员完成用户的完整使用流程 |
| **日常使用** | 将产品融入团队日常工作流程 |
| **发布门槛** | 用"对我们自己有用吗"作为发布标准 |
| **痛点记录** | 系统性记录使用中发现的问题 |

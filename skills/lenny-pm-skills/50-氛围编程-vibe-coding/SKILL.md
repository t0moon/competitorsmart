# 氛围编程

## 何时使用此技能

当你需要：
- 作为非技术人员快速构建功能原型
- 用自然语言而非代码来创建软件
- 展示概念验证而非静态设计稿
- 让PM或设计师能够直接构建和测试想法

## 核心原则

### 1. 从展示到构建
"氛围编程"是一种新的开发模式，非工程师（设计师/PM）使用AI构建功能软件和原型，无需深度掌握底层技术。

### 2. 自然语言即代码
通过自然语言和迭代反馈生成功能代码，而非手动编码。这超越了简单的提示工程。

### 3. 原型优于模型
用30分钟氛围编程出的功能原型来展示概念验证，比在Figma中展示静态设计更有说服力。

## 专家洞察

### Alexander Embiricos
> "他们会氛围编程一个动画编辑器，然后用这个动画编辑器来构建动画"

**核心洞察**：这个术语被反复使用来描述一种新的开发模式，非工程师（设计师/PM）使用AI构建功能软件和原型，无需深度掌握底层技术。

**如何应用**：
- 作为设计师或PM，使用AI工具构建功能原型
- 构建自己需要的工具，而非等待工程师实现

### Elena Verna
> "我自己就会氛围编程，所以我甚至会把它作为一项技能写在简历上。"

**核心洞察**：这被描述为非技术角色（PM、营销人员）使用自然语言构建功能软件的一项特定、变革性的技能，超越了简单的提示工程。

**如何应用**：
- 将氛围编程作为一项可展示的职业技能
- 用自然语言快速构建和测试想法

### Kevin Weil
> "我们不应该在Figma中展示东西，而应该展示人们在30分钟内氛围编程出来的原型，用来说明概念验证。"

**核心洞察**：一种新的快速原型方法，构建者使用AI通过自然语言和迭代反馈生成功能代码，而非手动编码。

**如何应用**：
- 用功能原型而非静态设计来展示想法
- 在30分钟内构建概念验证来验证想法

## 常见错误

- 继续用静态设计稿而非功能原型来沟通想法
- 等待工程师实现本可以快速原型验证的功能
- 低估非技术人员使用AI构建软件的能力
- 将氛围编程与简单的提示工程混为一谈

## 关键战术

| 战术 | 说明 |
|------|------|
| **原型优先** | 用功能原型而非静态设计来展示概念 |
| **30分钟验证** | 快速构建概念验证来测试想法可行性 |
| **自建工具** | 用AI构建自己需要的工具和编辑器 |
| **迭代对话** | 通过自然语言和AI的迭代对话来生成代码 |
| **技能展示** | 将氛围编程作为可展示的职业能力 |

## 工作目录与落盘规则

抓取到的内容通常很长，直接堆在上下文中会导致 token 溢出。所有 Phase 的抓取内容都遵循统一的"抓取即存盘 → 逐篇提炼 → 最终汇总"流程。

**工具说明：**
- 本文不绑定具体 MCP 名称
- 只要当前环境提供具备**网页搜索**与**网页抓取**能力的 MCP 即可
- 文中提到的"抓取 HTML"、"抓取 Markdown/纯文本"、"搜索引擎查询"，都可以替换为你当前环境中的等价 MCP 工具
- 可选使用具备**浏览器操作**能力的 MCP（如 Chrome / DevTools 类），用于点击、展开、翻页、切换 Tab、处理动态加载页面
- 如果当前环境没有网页搜索 MCP，则无法完整执行本流程

**工作目录（在 Phase 1 开始前创建）：**

```
{当前工作目录}/
└── {分析对象名}-资料/
    ├── raw/          # 原始抓取内容，每个文档一个 .md 文件
    ├── summary/      # 每个文档的结构化摘要
    └── merged.md     # 合并后的关键信息汇总（供撰写报告时引用）
```

1. **抓取即存盘**：每次抓取内容后，立即保存到 `raw/` 目录，文件名格式：`{序号}-{简要描述}.md`（如 `01-官网首页.md`、`05-招标文件.md`）。每个文件开头记录来源 URL 和抓取日期
2. **逐篇提炼**：对 `raw/` 中的每个文件，读取后提炼与分析维度相关的关键信息，丢弃无关内容。将提炼结果保存到 `summary/` 目录，与原文件同名。每份摘要应包含：
   - 文档来源与类型
   - 与分析维度相关的关键发现（用要点列表）
   - 关键数据/指标（如有）
   - 信息可信度评估（官方文档 > 第三方 > 未知来源）
3. **合并汇总**：三个 Phase 全部完成后，读取 `summary/` 中的所有摘要，合并为一份 `merged.md`，按分析维度（而非按文档来源）重新组织信息。这份文件是后续撰写报告的核心输入。

**注意事项：**
- 不要一次性把所有 raw 文件读入上下文，按文件逐一读取、提炼、保存摘要后再处理下一个
- 文档可能有时效性，优先选择日期较新的
- 部分文档可能是第三方（如高校、代理商）上传的，注意区分官方 vs 非官方来源
- 如果抓取的文档内容为乱码或无法解析，跳过即可，不要在无效文档上浪费时间
- 招投标文件特别有价值——它们通常包含竞品不会在官网公开的真实定价和技术参数

## Phase 1: 官网信息采集

**这是信息收集的起点。** 先摸清官网的完整结构，再根据结构有目的地深入各页面，而不是盲目猜测页面路径。

### 第一步：抓取首页，摸清网站结构

1. **定位官网**：搜索 `"{产品名} 官网"` 找到产品/公司官方网站
2. **抓取首页 HTML**：使用具备网页抓取能力的 MCP，以 HTML 形式抓取官网首页，重点关注：
   - 顶部导航栏 / 菜单结构（`<nav>`、`<header>` 区域的链接）
   - 底部链接（`<footer>` 区域通常包含"关于我们"、"帮助中心"、"隐私政策"等入口）
   - 页面内嵌的核心跳转链接（如"了解更多"、"查看详情"、"免费试用"等 CTA 按钮指向的 URL）
3. **提取链接清单**：从首页 HTML 中整理出网站的完整导航结构，形成一张链接地图：
   ```
   官网首页: https://example.com
   ├── 产品介绍: https://example.com/products
   ├── 解决方案: https://example.com/solutions
   ├── 定价: https://example.com/pricing
   ├── 关于我们: https://example.com/about
   ├── 新闻动态: https://example.com/news
   ├── 帮助中心: https://help.example.com
   └── ...
   ```
4. **将首页内容保存到 `raw/01-官网首页.md`**，同时再抓取一份 Markdown 或纯文本版本用于内容分析

**可选补充：使用浏览器操作类 MCP**
- 如果首页导航被折叠在菜单中，先展开菜单再提取链接
- 如果页面内容依赖 Tab 切换、点击"查看更多"、滚动加载或分页，使用浏览器操作类 MCP 逐步展开后再抓取
- 如果纯抓取工具拿不到完整 DOM 或文本，再切换到浏览器操作类 MCP 获取页面快照或展开后的内容

### 第二步：根据链接地图逐页深入

根据上一步得到的链接清单，有选择地抓取与分析维度相关的页面。使用具备网页抓取能力的 MCP，以 Markdown 或纯文本形式抓取各页面内容。优先级如下：

| 优先级 | 页面类型 | 分析价值 |
|--------|---------|---------|
| 必抓 | 产品/功能介绍页 | 产品矩阵、核心功能、差异化卖点 |
| 必抓 | 关于我们/公司介绍页 | 公司背景、团队、发展历程 |
| 必抓 | 定价/方案页 | 商业模式、收费结构、客户分层 |
| 建议抓 | 解决方案/行业方案页 | 目标行业、应用场景、客户案例 |
| 建议抓 | 新闻/动态页 | 最新战略方向、产品更新 |
| 建议抓 | 帮助中心/文档中心入口 | 功能深度、文档资源线索 |
| 按需抓 | 合作伙伴/生态页 | 生态布局、合作关系 |
| 按需抓 | 招聘页 | 团队规模、技术栈、扩张方向 |

- **产品功能分析优先建议**：如果能找到产品实际入口（如 Web App、控制台、演示环境、公开体验页），建议优先使用浏览器操作类 MCP 进入产品内部，沿真实用户路径体验和记录功能，而不是只依赖官网文案。
- 每个页面抓取后立即保存到 `raw/`，文件名如 `02-产品介绍.md`、`03-关于我们.md`
- 如果某个页面内还有子导航或"查看更多"的链接，且内容与分析维度相关，继续深入一层抓取
- 不要无限递归——一般深入到第二层就够了，除非用户指定了需要特别深入的方向
- 遇到单页应用、懒加载、折叠面板或需要用户交互才能看到的内容时，可选使用浏览器操作类 MCP 辅助采集

### 第三步：建立初步认知

阅读已抓取的官网信息，提炼出：
- 产品矩阵和核心功能模块
- 公司定位和目标客户
- 官方宣传的差异化优势
- **信息空白点**：哪些维度官网没有覆盖或信息不足？（这些空白将指导 Phase 2 的搜索方向）

**输出物：** 官网信息摘要保存到 `summary/00-官网概览.md`，包含网站结构地图和信息空白点清单。

## Phase 2: 定向扩展搜索

根据官网信息中发现的空白和线索，有目的地进行扩展搜索。搜索方向由 Phase 1 的认知驱动，而不是漫无目的地搜。

**常见扩展方向：**
- 官网提到但未详述的功能/技术 → 搜索 `"{产品名} {功能名} 详解/评测"`
- 官网未公开的财务/规模数据 → 搜索 `"{公司名} 营收/融资/市场份额"`
- 官网未提及的用户评价 → 搜索 `"{产品名} 评价/口碑/使用体验"`
- 行业地位和竞争环境 → 搜索 `"{行业} 市场格局/排名"`
- 聚焦维度的深度信息 → 搜索 `"{产品名} {聚焦维度关键词}"`

**搜索工具：**
- 使用具备网页搜索能力的 MCP 搜索行业报告、公司信息、产品数据
- 使用具备网页抓取能力的 MCP 抓取有价值的页面详细信息
- 必要时可选使用浏览器操作类 MCP 打开结果页、展开隐藏内容或处理动态页面
- 并行搜索以提高效率
- **搜索范围约束**：只搜索用户确认的分析对象，如果搜索过程中发现可能遗漏的重要竞品，告知用户由用户决定是否加入，不自行加入

将有价值的搜索结果保存到 `raw/` 并提炼摘要到 `summary/`。

### 功能清单审查与逐项深挖

Phase 1 和 Phase 2 的搜索过程中，经常会在搜索结果摘要、文章段落、高校通知等处**顺带提到**分析对象的各种功能/产品/技术名词，但只有名称没有细节（例如只看到"AI教案"三个字，但不知道它具体做什么、怎么用、效果如何）。如果就此止步，最终报告中对这些功能只能一笔带过，导致分析深度不够。

**操作步骤：**

1. **汇总功能清单**：回顾已收集的所有 `raw/` 和 `summary/` 文件，将提到的所有功能/产品/技术模块整理为一张清单：

   ```
   功能/模块名称        | 当前信息深度    | 是否需要深挖
   AI助教              | 有详细说明      | ❌ 已充分
   AI陪练              | 有使用手册描述   | ❌ 已充分
   AI教案              | 仅提及名称      | ✅ 需要深挖
   数字人平台           | 仅知道名称      | ✅ 需要深挖
   智能体广场           | 仅提及概念      | ✅ 需要深挖
   ```

2. **逐项深挖**：对标记为"需要深挖"的功能，进行针对性搜索：
   - 搜索 `"{产品名} {功能名}" 功能 介绍 OR 使用 OR 教程 OR 评测`
   - 搜索 `"{产品名} {功能名}" 使用手册 OR 操作指南`
   - 如果功能名较新或较冷门，尝试搜索相关高校的使用报道：`"{产品名} {功能名}" site:edu.cn`
   - 如果已经找到产品入口，优先使用浏览器操作类 MCP 实际进入产品，查看菜单结构、功能入口、配置项、交互路径和输出结果
   - 对搜索到的有价值页面，用具备网页抓取能力的 MCP 抓取详情

3. **判断深挖到什么程度算"够用"**：
   - 能回答"这个功能**是什么、怎么用、解决什么问题**"→ 基本够用
   - 如果还关注聚焦维度，还需要回答"**做到什么程度、和同类方案比有什么差异**"→ 深度够用
   - 搜了 2-3 轮仍然只有名称没有细节 → 标注"信息未公开，仅知功能存在"，不再耗时

4. **更新摘要**：将深挖到的功能细节补充到对应的 `summary/` 文件中，或新建专题摘要文件

**注意**：这一步只针对与分析维度相关的功能，不是所有功能都要深挖。如果用户聚焦的是"AI模块"，那只需要深挖 AI 相关的功能；与聚焦维度无关的功能只需保留名称级别的记录即可。

经过 Phase 1 和 Phase 2，已经对分析对象有了基本和扩展认知。这一阶段针对仍然不够深入的维度，通过挖掘文档类资源获取官网和公开报道中看不到的细节。

### 搜索方法

在搜索关键词后加上文件格式后缀，挖掘搜索引擎索引到的各类文档：

```
"{产品名} {关键词} filetype:pdf"
"{产品名} {关键词} filetype:pptx"
"{产品名} {关键词} filetype:docx"
"{产品名} {关键词} filetype:txt"
```

**重点挖掘的文档类型：**

| 文档类型 | 搜索关键词示例 | 价值 |
|---------|--------------|------|
| 产品使用手册 | `"{产品名} 使用手册 filetype:pdf"` | 完整功能列表、操作流程 |
| 帮助文档 | `"{产品名} 帮助文档 操作指南 filetype:pdf"` | 功能细节、技术实现 |
| 培训/宣讲材料 | `"{产品名} 培训 方案 filetype:pptx"` | 产品卖点、市场定位、客户案例 |
| 招投标文件 | `"{产品名} 招标 采购 filetype:pdf"` | 真实定价、技术参数、交付标准 |
| 技术白皮书 | `"{产品名} 白皮书 技术方案 filetype:pdf"` | 技术架构、性能指标 |
| 行业研究报告 | `"{产品名} 研究报告 分析 filetype:pdf"` | 第三方评价、市场数据 |
| 客户案例/合同 | `"{产品名} 案例 实施 filetype:docx"` | 实际应用场景、客户反馈 |

### 文件下载与转换

PDF、DOCX、PPTX 等二进制文件通常无法通过网页抓取 MCP 直接读取，必须先下载到本地再转换为 Markdown。

**第一步：下载文件**

从搜索结果中获取文件的直接 URL，使用 `curl` 下载到 `raw/` 目录：
```bash
curl -L -o "{分析对象名}-资料/raw/{序号}-{描述}.{ext}" "{文件URL}"
```
注意加 `-L` 跟随重定向。如果 URL 需要编码，用引号包裹。

**第二步：转换为 Markdown**

根据文件类型选择转换方式：

| 文件类型 | 首选方案 | 备选方案（首选不可用时） |
|---------|---------|----------------------|
| PDF | `pdftotext -layout "{路径}" "{输出}.txt"` 然后重命名为 `.md`（需 `brew install poppler`） | Python: `pip install pdfplumber` 后用 `pdfplumber.open()` 提取文本 |
| DOCX | `pandoc "{路径}" -t markdown -o "{输出}.md"`（需 `brew install pandoc`） | Python: `pip install python-docx` 后用 `Document()` 提取段落文本 |
| PPTX | Python: `pip install python-pptx` 后用 `Presentation()` 逐 slide 提取文本 | — |
| TXT | 直接 `cp` 并重命名为 `.md` | — |

**第三步：读取并提炼**

用 Read 工具读取转换后的 `.md` 文件，然后按常规落盘规则提炼摘要到 `summary/`。

**转换失败的处理：**
- 转换工具未安装 → 先尝试安装（`brew install` 或 `pip install`）
- 安装也失败 → 跳过，在摘要中标注"文件格式无法解析，需要用户手动提供内容"
- 加密或受保护的文件 → 跳过并标注
